# ტექნიკური დავალება (TRD): RustDesk x Academy Support Portal

## 1. პროექტის მიმოხილვა

**მიზანი:** შეიქმნას სწრაფი და მარტივი პლატფორმა აკადემიის სტუდენტებისთვის, რათა მენტორებმა/ადმინებმა შეძლონ მათი დისტანციური დახმარება RustDesk-ის გამოყენებით.
**მთავარი ფასეულობა:** სტუდენტისთვის მაქსიმალურად გამარტივებული პროცესი (ავტორიზაციის გარეშე), რეალურ დროში (Real-time) კომუნიკაცია და დახმარების სესიების ეფექტური მართვა.

## 2. სისტემის არქიტექტურა

სისტემა შედგება სამი ძირითადი კომპონენტისგან:

- **Client (სტუდენტის აპლიკაცია):** მსუბუქი Web აპლიკაცია (Next.js), რომელიც მუშაობს `localStorage`-ზე.
- **Admin Panel:** დაცული Web პანელი მენტორებისთვის/ადმინებისთვის.
- **API & WebSockets (Backend):** სერვერი (Node.js), რომელიც უზრუნველყოფს მონაცემთა შენახვას და WebSocket (Socket.io) კავშირებს რეალურ დროში სტატუსების განსაახლებლად.

## 3. მომხმარებელთა როლები და ფუნქციონალი

### 3.1 ადმინისტრატორი (Admin)

- **ავტორიზაცია:** სისტემაში შესვლა ლოგინით და პაროლით (JWT).
- **ჯგუფების მართვა:** ახალი ჯგუფის შექმნა და უნიკალური, მარტივი კოდის (მაგ. `JS-101`, `REACT-55`) გენერაცია.
- **მონიტორინგი რეალურ დროში:** თითოეული ჯგუფის შიდა გვერდზე აქტიური (დაჯოინებული) სტუდენტების სიის ხილვა WebSocket-ის დახმარებით.
- **დახმარების სესიის მართვა:** \* ხედავს სტუდენტებს, რომლებმაც დააჭირეს "Help" ღილაკს.
  - ხედავს სტუდენტის **Full Name, RustDesk ID, RustDesk Password**.
  - **"Connect" ღილაკი:** მასზე დაჭერით იხსნება RustDesk აპლიკაცია (Custom URI Scheme-ის ან ბრაუზერის სხვა მექანიზმის საშუალებით) და სტუდენტის სტატუსი იცვლება "Ongoing"-ზე.
  - **"Complete" ღილაკი:** დახმარების დასრულების შემდეგ, სესიის დახურვა და სტუდენტის სტატუსის განახლება.

### 3.2 სტუდენტი (Client)

- **ავტორიზაცია:** არ მოითხოვება. იდენტიფიკაცია ხდება Local Storage-ით და Session ID-ით.
- **Onboarding / Join:** საიტის გახსნისას ხვდება ველი: "შეიყვანეთ ჯგუფის კოდი".
- **ინფორმაციის შეყვანა:** კოდის ვალიდაციის შემდეგ შეჰყავს: Full Name, RustDesk ID, RustDesk Password.
- **პროფილის გვერდი:**
  - ინახავს მონაცემებს `localStorage`-ში (თუ გვერდს დაარეფრეშებს, მონაცემები ადგილზეა).
  - შეუძლია მონაცემების რედაქტირება.
  - **"Ask for Help" ღილაკი:** აგზავნის სიგნალს ადმინთან.
  - **სტატუსის ინდიკაცია:** როცა ადმინი იწყებს დახმარებას (Ongoing), ღილაკი ხდება არააქტიური და გამოდის შეტყობინება: "მენტორი უკავშირდება თქვენს კომპიუტერს...". დასრულების შემდეგ ღილაკი ისევ აქტიურდება.

## 4. პროცესების ნაკადი (User Flow)

1.  **ჯგუფის შექმნა:** ადმინი ქმნის ჯგუფს ➔ გენერირდება კოდი ➔ აზიარებს კოდს სტუდენტებთან.
2.  **დაკავშირება:** სტუდენტი შედის საიტზე ➔ წერს კოდს ➔ ავსებს RustDesk-ის მონაცემებს ➔ ხვდება ლოდინის რეჟიმში (პროფილის გვერდი). მონაცემები ინახება ბრაუზერის `localStorage`-ში.
3.  **დახმარების მოთხოვნა:** სტუდენტი აჭერს "დახმარებას". WebSocket-ით იგზავნება ივენთი: `request_help`.
4.  **სესიის დაწყება:** ადმინის პანელში სტუდენტი ინაცვლებს "Needs Help" სიაში. ადმინი აჭერს დასახმარებელ ლინკს. იგზავნება ივენთი: `status_ongoing`. სტუდენტის ეკრანზე ღილაკი იბლოკება.
5.  **დასრულება:** ადმინი ასრულებს საქმეს და აჭერს "Complete". იგზავნება ივენთი: `status_idle`. სტუდენტს კვლავ შეუძლია დახმარების მოთხოვნა.

## 5. WebSockets სტრუქტურა (Event-ები)

რეალურ დროში სინქრონიზაციისთვის დაგჭირდება შემდეგი ძირითადი ივენთები (Events):

- `join_group`: სტუდენტი უერთდება კონკრეტულ ჯგუფს (Socket Room).
- `student_updated`: როცა სტუდენტი აკეთებს საკუთარი ინფოს (ID/Password) რედაქტირებას, ადმინთან მონაცემები ლაივში ახლდება.
- `help_requested`: სტუდენტიდან ➔ ადმინისკენ.
- `help_started`: ადმინიდან ➔ სტუდენტისკენ (ცვლის UI სტატუსს Ongoing-ზე).
- `help_completed`: ადმინიდან ➔ სტუდენტისკენ (აბრუნებს UI-ს საწყის მდგომარეობაში).
- `student_disconnected`: თუ სტუდენტმა დახურა ტაბი, ადმინთან მისი სტატუსი უნდა გაშავდეს (Offline) ან გაქრეს სიიდან.

## 6. რჩევები და ტექნიკური რისკები (Risks & Mitigations)

1.  **RustDesk Deep Linking:**
    - _რისკი:_ a (anchor) ტეგის საშუალებით აპლიკაციის გახსნა და მონაცემების (ID/Password) პირდაპირ გადაცემა. უნდა გადაამოწმო, აქვს თუ არა RustDesk-ს მხარდაჭერილი Custom URI scheme (მაგალითად, `rustdesk://<id>?password=<pass>`).
    - _გადაწყვეტა:_ თუ პაროლის პირდაპირ ლინკით გადაცემა არ ხერხდება უსაფრთხოების ან აპლიკაციის შეზღუდვის გამო, გააკეთე ისე, რომ ადმინის ღილაკზე დაჭერისას ID გაიხსნას აპლიკაციაში, ხოლო პაროლი ავტომატურად დაკოპირდეს (Clipboard) ადმინის მაუსზე, რათა მარტივად ჩასვას (Ctrl+V).
2.  **უსაფრთხოება (Security):**
    - მნიშვნელოვანია იცოდე, რომ `localStorage`-ში პაროლის შენახვა ოდნავ სარისკოა, თუ კომპიუტერი საერთოა. თუმცა, რადგან ეს დროებითი სასწავლო პროცესია, მისაღებია.
    - **აუცილებელია:** API-სა და WebSocket-ებს შორის ინფორმაციის (განსაკუთრებით RustDesk პაროლების) გაცვლა ხდებოდეს მხოლოდ დაცული პროტოკოლით (**HTTPS / WSS**), რათა ქსელში მონაცემები არ გადაიცეს ღიად.
3.  **Local Storage Management:**
    - დაამატე ფუნქცია "Leave Group" ან "Clear Data" სტუდენტის პროფილზე, რათა სესიის/კურსის დასრულების შემდეგ შეეძლოს თავისი მონაცემების წაშლა ბრაუზერიდან.
